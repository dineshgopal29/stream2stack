Wiki Knowledge Base

Stream2Stack Phase B In Progress Version 0.2 Updated 2026-04-05

Overview

The Wiki Knowledge Base treats LLM-powered content synthesis as a build system. Raw video transcripts are "source files"; the compiler turns them into structured, auditable Markdown pages; backlinks wire them into a navigable knowledge graph.

Page Schema

Each wiki page is a .md file with YAML frontmatter. The schema is intentionally simple — replaceable by a Supabase table in a later phase.

---
title: "Retrieval-Augmented Generation"
slug: retrieval-augmented-generation
type: concept          # concept | tool | pattern | index | health | qa_note
schema_version: 1
compiled_at: 2026-04-05T10:00:00+00:00
source_hash: a3f1b2c9...
sources:
  - <video-uuid-1>
  - <video-uuid-2>
backlinks:
  - "Vector Database"
  - "Embedding"
---

## Summary
...

Incremental recompile logic

  • Page missing → recompile
  • schema_version < SCHEMA_VERSION constant → recompile (schema migration)
  • source_hash changed (new video added) → recompile
  • Otherwise → skip

Phase B — Q&A with Artifact Filing

Wiki Query Service

In Progress

Allow developers to ask free-form questions against the wiki. Answers are grounded in compiled pages, filed back as qa_notes, and returned with source citations.

New endpoint

POST /wiki/query
{ "question": "What are the tradeoffs between RAG and fine-tuning?", "user_id": "alice" }

→ {
    "answer": "...",
    "sources": ["concepts/retrieval-augmented-generation", "concepts/fine-tuning"],
    "qa_note_slug": "what-are-the-tradeoffs-between-rag-and-fine-tuning",
    "pages_searched": 6
  }

Implementation flow (wiki_query.py)

  1. Load all compiled wiki pages from the store.
  2. Score pages by keyword overlap with the question (no embeddings — fast and deterministic).
  3. Pass top-8 pages (≤ 1,500 chars each) as context to the LLM.
  4. Extract ## Sources block from LLM response; parse slug paths.
  5. Write qa_note to local_storage/wiki/qa_notes/<slug>.md.
  6. Return answer + sources + qa_note_slug.

Phase C — Linter (Planned)

Health Checker

Planned

Scheduled health checks detect structural issues in the wiki:

  • Orphan concepts — mentioned in backlinks but no page exists
  • Missing code examples — pages without a ## Code Example section
  • Stale pages — compiled_at older than 30 days with new video activity
  • Contradiction candidates — same term, conflicting definitions across sources
  • Broken backlinks — [[Term]] references that have no corresponding page

Output: local_storage/wiki/health/report-<date>.md
Triggered via: GET /wiki/health or scheduled cron.
